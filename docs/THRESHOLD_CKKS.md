# Threshold CKKS Architecture for Opaque

## The Key Ownership Problem

Opaque's current architecture generates a single CKKS secret key in the client process. Whoever holds this key can decrypt all HE computation results, including query-centroid similarity scores. In a real deployment where the server performs HE computation on behalf of multiple clients, this creates a fundamental trust problem: if the server holds the key, it can see everything. If only one client holds the key, other clients can't use the system.

This document describes the planned architecture for solving this using threshold CKKS homomorphic encryption, where the decryption key is split across multiple independent nodes so that no single party can decrypt.

## Background: How Threshold CKKS Works

### Single-Party CKKS (Current)

A CKKS ciphertext is a pair (c0, c1). Decryption recovers the plaintext as:

    plaintext = c0 + c1 * sk

One key, one decryptor, one point of failure.

### Threshold CKKS

The secret key sk is never generated as a whole. Instead, N independent parties each generate a random key share locally:

    Party 1: share_1 (random)
    Party 2: share_2 (random)
    ...
    Party N: share_N (random)

The "ideal" secret key is implicitly sk = share_1 + share_2 + ... + share_N, but it is never materialized anywhere.

Because decryption is linear in the secret key:

    plaintext = c0 + c1 * (share_1 + share_2 + ... + share_N)
              = c0 + (c1 * share_1) + (c1 * share_2) + ... + (c1 * share_N)
              = c0 + partial_1 + partial_2 + ... + partial_N

Each party computes their partial independently. The combination is addition. No party ever sees the full key. No party ever sees another party's share.

### t-of-N Threshold via Shamir Secret Sharing

For availability, we don't want to require all N parties for every decryption. Using Shamir secret sharing, the key shares are distributed such that any t parties (the threshold) can jointly decrypt, but t-1 parties learn nothing.

Lattigo's mhe package provides `Thresholdizer` for Shamir distribution and `Combiner` for converting t-of-N Shamir shares into t-of-t additive shares compatible with the decryption protocol.

### Noise Flooding

Each partial decryption share is "flooded" with extra noise before being sent. This prevents the partial from leaking information about the party's key share. The noise is large enough to statistically mask the share but small enough to preserve the plaintext. With CKKS parameters like Opaque's (LogN=14, deep modulus chain), there is ample headroom for noise flooding without affecting result precision. Centroid similarity scores (values in [-1, 1]) are not affected at any meaningful decimal place.

## Architecture

### Roles

**DB Server (1 node)**
- Stores the encrypted vector index (AES blobs, cluster structure)
- Holds CKKS evaluation keys (public key, relinearization key, Galois keys) but NO secret key material
- Performs all HE computation (dot products between encrypted queries and plaintext-encoded centroids)
- Coordinates the threshold decryption protocol by collecting and aggregating partial key-switches from the committee
- Returns encrypted results to the client
- Cannot decrypt anything, even if fully compromised

**Key Committee (N nodes, threshold t)**
- Each node stores exactly one key share (a few KB)
- On request: computes a partial key-switch on a ciphertext and returns it
- Lightweight: no vector storage, no search computation, minimal CPU per query
- Can be run by different organizations, in different regions, on different cloud providers
- Any t nodes can service a decryption; the remaining N-t can be offline

**Client**
- Generates an ephemeral CKKS keypair (client_sk, client_pk) per session or per query
- Encrypts the query vector using the collective public key (the one generated by the committee during setup)
- Sends (encrypted_query, client_pk) to the DB server
- Receives a single ciphertext encrypted under client_pk
- Decrypts locally with client_sk to get centroid similarity scores
- Selects top-K clusters, fetches AES-encrypted blobs, decrypts and reranks locally

### Protocol: Setup (One-Time, Offline)

Setup requires all N key committee nodes to be online simultaneously. This happens once (or on key rotation events).

**Step 1: Local Key Share Generation**

Each committee node i independently generates a random CKKS secret key share using Lattigo's standard key generator. No communication needed.

**Step 2: Threshold Distribution (Optional, for t-of-N)**

If t < N is desired, each node creates a degree-(t-1) Shamir polynomial with their share as the constant term, evaluates it at each other node's public point, and sends the evaluation over a private channel. Each node aggregates received evaluations to get their Shamir share.

This step requires private pairwise channels between all nodes.

**Step 3: Collective Public Key Generation (1 round)**

Each node generates a public key share from their secret key share using a common random string (CRS). Shares are sent to a coordinator (can be the DB server), which aggregates them into the collective public key. The CRS is derived from a shared seed, so all nodes produce compatible shares.

**Step 4: Collective Relinearization Key Generation (2 rounds)**

This is the only multi-round protocol. Each node generates a round-1 share plus an ephemeral secret key. Round-1 shares are aggregated. Each node then uses the aggregated round-1 result plus their ephemeral key to generate a round-2 share. Round-2 shares are aggregated to produce the relinearization key.

The relinearization key enables the server to perform HE multiplications (needed for dot products). It contains no secret key material.

**Step 5: Collective Galois Key Generation (1 round per rotation)**

Galois keys enable CKKS slot rotations (needed for summing dot product components). Each required rotation step needs one collective key generation round. For Opaque's batch dot product, this is approximately log2(dimension) rotations, so about 7 rounds for 128-dimensional vectors.

**Setup Output**

The DB server receives and stores:
- Collective public key (used to encrypt queries)
- Relinearization key (used for HE multiplication)
- Galois keys (used for HE rotation)

None of these allow decryption.

### Protocol: Query (Per Search Request)

**Step 1: Client Encrypts Query**

The client generates a fresh ephemeral keypair (client_sk, client_pk). It encrypts the query vector using the collective public key and sends both the encrypted query and client_pk to the DB server.

Why the collective public key and not client_pk? Because the centroids were pre-encoded against the collective key's parameters, and HE operations require operands under the same key.

**Step 2: DB Server Performs HE Search**

The DB server computes HE dot products between the encrypted query and all plaintext-encoded centroids. This is identical to the current Opaque search path. No changes needed.

The result is a ciphertext (or batch of ciphertexts) containing encrypted similarity scores, encrypted under the collective key.

**Step 3: DB Server Fans Out to Key Committee**

The DB server sends the result ciphertext and the client's public key to the key committee nodes (or at least t of them).

**Step 4: Each Committee Node Computes a Partial Key-Switch**

Each node i uses Lattigo's PublicKeySwitchProtocol to compute a partial re-encryption from the collective key toward the client's public key. This operation uses the node's key share and adds noise flooding.

The partial is not a plaintext fragment. It is a polynomial in the CKKS ring that, when combined with other partials, produces a valid CKKS ciphertext under client_pk. An individual partial reveals nothing about the plaintext or the key share (due to noise flooding).

**Step 5: DB Server Aggregates Partials**

The DB server collects t partial key-switch shares and aggregates them (polynomial addition). The result is a CKKS ciphertext encrypted under client_pk.

The DB server performs this aggregation but the result is still encrypted. The server does not have client_sk. It cannot decrypt. It is safe for the server to handle all partials and perform the aggregation because the individual partials reveal nothing and the aggregate is encrypted under the client's key.

**Step 6: DB Server Returns Result to Client**

The client receives a single CKKS ciphertext encrypted under their own public key. They decrypt it locally with client_sk to obtain centroid similarity scores.

**Step 7: Standard Opaque Flow Continues**

The client selects top-K clusters, adds decoy clusters, requests AES-encrypted blobs, decrypts them locally, reranks, and returns results. This is unchanged from the current architecture.

### Why the DB Server Cannot Cheat

The DB server sees:
- The encrypted query (can't decrypt: no secret key)
- The result ciphertext under the collective key (can't decrypt: no secret key)
- The partial key-switch shares from each committee node (each is noise-flooded, reveals nothing individually)
- The aggregated result under client_pk (can't decrypt: no client_sk)

Even with full access to all network traffic and all stored data, the server learns nothing about the query, the scores, or which clusters were selected.

### Why a Committee Node Cannot Cheat

Each committee node sees:
- The ciphertext it's asked to partially key-switch (encrypted, can't decrypt with just one share)
- The client's public key (public information)
- Its own partial result (noise-flooded, reveals nothing about the plaintext)

A single compromised node learns nothing. Even t-1 compromised nodes learn nothing (the Shamir threshold guarantee).

## Deployment Models

### Model 1: Enterprise Key Committee

A single organization deploys Opaque and runs the key committee across its own infrastructure. Nodes are placed in different availability zones, cloud accounts, or regions. Security teams manage the shares.

Best for: organizations that want to guarantee "no single server compromise breaks privacy" while keeping operational control. The committee nodes are lightweight (minimal compute, a few KB of storage) so the cost is low.

Latency impact for same-region deployment: approximately 7-20ms overhead per query. For cross-region (e.g., US-East + EU-West + Asia): approximately 100-300ms overhead, dominated by the round-trip to the farthest node. Mitigation: query more than t nodes in parallel, use the first t responses.

### Model 2: Multi-Stakeholder Consortium

Different organizations each hold key shares. No single organization can decrypt. Queries require cooperation from t organizations.

Example: A hospital stores patient embedding vectors. The hospital runs the DB server. Key shares are held by: the hospital's security team, an independent compliance auditor, and the insurance partner. Any 2 of 3 can authorize decryption.

Best for: regulated industries (healthcare, finance, legal) where multi-party oversight is required by compliance.

### Model 3: Client-Contributed Committee

Each client organization that uses the vector DB also contributes a key committee node. The threshold is set proportional to the number of participants.

Best for: federated or consortium deployments where multiple organizations share a private vector database and no single organization should have disproportionate control.

### Model 4: Independent Key Escrow Services

Third-party key management services each hold a share. The data owner pays escrow fees. This mirrors the custody model in cryptocurrency.

Best for: a startup or SaaS product where customers want provable privacy without running their own infrastructure.

## Node Integrity: Detecting and Ejecting Bad Nodes

### The Problem

A malicious committee node could send garbage partial decryptions, causing incorrect results. Under the passive (semi-honest) security model, we assume nodes follow the protocol. But in production, we want detection and recovery.

### Approach 1: Redundancy Checks (Recommended for v1)

Query all N nodes instead of just t. Pick two different subsets of t nodes, combine their partials independently, and compare the decrypted results. If they match, all nodes are honest. If they diverge, use subset intersection to identify the bad node.

Cost: 2x the partial decryption shares (which are cheap), zero cryptographic overhead.

### Approach 2: Verifiable Secret Sharing (VSS)

During setup, each node commits to their key share using Pedersen commitments. Later, anyone can verify that a node's partial decryption is consistent with their committed share. This is a standard technique with moderate implementation effort.

### Approach 3: ZK Proofs of Correct Partial Decryption

Each node produces a zero-knowledge proof that their partial was computed correctly. This provides the strongest guarantee but lattice-based ZK proofs are expensive (100ms-1s per proof). This is an active research area and a potential future enhancement.

### Strike-Based Ejection

Regardless of which detection approach is used, a simple operational policy handles bad nodes:

1. On each query, run a redundancy check
2. If a node's partial is inconsistent, record a strike
3. After K strikes (e.g., 3), the node is removed from the active committee
4. Trigger a key re-sharing protocol to distribute shares among the remaining N-1 nodes

This doesn't require consensus or voting, just consistent observation by the coordinator (DB server) or the client.

## Implementation Status

Threshold CKKS is fully implemented and integrated as an **optional** backend.

### Packages

- **`pkg/crypto/threshold/`** — Core threshold CKKS: `Committee`, `Participant`, `ClientSession`. Wraps Lattigo's `mhe` package for collective key generation, Shamir threshold distribution, and `PublicKeySwitchProtocol`.

- **`pkg/crypto/provider.go`** — `HEProvider` interface that abstracts the HE backend. Two implementations:
  - `DirectHEProvider` (default): wraps `EnginePool`, single-key mode.
  - `ThresholdHEProvider`: wraps `Committee` + `ClientSession`, t-of-N threshold mode.

- **`pkg/client/enterprise_hierarchical.go`** — Uses `HEProvider` interface. Existing constructors default to direct mode. `NewEnterpriseHierarchicalClientWithProvider()` accepts any `HEProvider`.

### What Changed

The `HEProvider` interface replaces direct `EnginePool` usage in the client. This is a clean abstraction — the client calls `provider.EncryptVector()`, `provider.Acquire()`, `engine.DecryptScalar()` etc. without knowing whether decryption is local or goes through a t-of-N protocol.

For threshold mode, `ThresholdHEProvider` creates:
1. A pool of evaluator-only engines using the committee's collective eval keys (relin + Galois).
2. An internal `ClientSession` with an ephemeral keypair for receiving threshold-decrypted results.
3. Decryption calls delegate to `committee.ThresholdDecrypt(ct, session.PK)` then `session.DecryptScalar(ctClient)`.

### What Didn't Change

- The HE dot product computation (the core of search) — identical code path.
- AES blob encryption/decryption (client-side, unchanged).
- The decoy mechanism (client-side, unchanged).
- K-means clustering and index building.
- The batch CKKS slot packing optimization.
- Incremental indexing.
- All existing tests — `NewClientEngine()` and `EnginePool` still exist and work.

### Parallelized Threshold Decryption

`ThresholdDecrypt` is now parallelized internally: Shamir share conversion (Shamir-to-additive) and PCKS partial share generation run in goroutines per participant, simulating a real distributed deployment where each committee node computes independently. This required giving each `thresholdEvalEngine` its own decryptor and encoder — Lattigo's `rlwe.Decryptor` is NOT thread-safe, and sharing one across goroutines caused a data race that manifested as underflow panics during concurrent decryption. The unnecessary mutex on the `Committee` struct was also removed.

##### Cross-Dataset Summary

**Threshold overhead is ~0-10% at scale** across SIFT100K, SIFT1M, and GIST100K. At higher absolute latencies (especially high-dimensional GIST at 960-dim), threshold decryption cost becomes completely invisible.

**Recall is equivalent between direct and threshold modes** in all configurations across all three datasets. Noise flooding in the threshold protocol does not degrade search quality.

**Sweet spot (SIFT 100K):** probe-32 (50%) gives 97.8% recall@10 at 368ms direct / 322ms threshold.

**Sweet spot (SIFT1M):** probe-48 (37.5%) gives 95.0% recall@10 at 653ms direct / 719ms threshold — meeting the ~95% recall target with sub-second latency. probe-64 (50%) pushes to 99.2%/99.0% recall at ~860ms.

**SIFT1M at 100% recall:** probe-96 (75%) achieves perfect recall at ~1.1s direct / ~1.14s threshold — only 1.06x overhead.

**SIMD slot packing:** With 8192 usable CKKS slots and 128-dim vectors, 64 centroids are packed into a single ciphertext. All 64 centroids are scored in 1 HE multiply + 7 rotations (log2(128)). SearchBatch reduces what would be 64 separate HE operations to just 1.

The per-op micro-benchmarks above (encrypt=10ms, dot=119ms, decrypt=24ms for 3-of-5) remain valid for individual operations.

## Noise and Recall Impact

Two noise sources exist in the threshold setting:

**CKKS approximation noise** (already present): every HE operation adds noise. Opaque's shallow circuits (dot product = multiply + log2(d) rotations + additions) consume a small fraction of the available noise budget. Current precision is approximately 15-17 significant bits for normalized similarity scores.

**Noise flooding in partial decryption** (new): each partial is flooded with approximately 40-50 bits of Gaussian noise to mask the key share. With Opaque's parameter set providing 200+ bits of total modulus chain, and dot products consuming roughly 60-80 bits, there remains 120+ bits of headroom.

50 bits of flooding into 120+ bits of headroom means no meaningful precision loss. Centroid similarity scores (values in [-1, 1]) that differ by magnitudes of 0.01-0.1 are entirely unaffected. Top-K cluster selection will produce identical results to the single-party setting.

Lattigo provides utility functions (NoiseKeySwitch) that compute exact noise overhead for given parameters and party counts. These should be validated during implementation.

## Latency Overhead Summary

Per-query overhead from threshold decryption, by deployment topology:

Same datacenter: The DB server fans out to committee nodes in parallel. Network round-trip is 1-2ms, partial computation is 5-15ms, return is 1-2ms. Total overhead approximately 10-20ms. With current Opaque search latency of 200-500ms, this is a 2-10% increase.

Same region, different availability zones: Network round-trip 5-10ms. Total overhead approximately 15-30ms.

Cross-region (e.g., committee spread across US, EU, Asia): Network dominated by farthest node, 50-150ms each way. Total overhead approximately 100-300ms. Mitigated by querying more than t nodes and using the first t responses.

Global worst case (nodes on opposite sides of the planet): approximately 300ms round-trip. With t=3 and N=5, query all 5 nodes in parallel, use first 3 responses. Effective overhead is the 3rd-fastest round-trip, likely 150-200ms.

Setup (one-time) takes longer due to multi-round protocols (especially relin key gen), but this happens once and is not latency-sensitive.

## Comparison with MPC (PRAG) Approach

The PRAG paper (Zyskind, South, Pentland, 2023) solves the same problem using MPC instead of HE:

In PRAG, the data itself is secret-shared across multiple servers. Every computation (distance calculation, top-k selection) requires interactive MPC protocols with O(k * log N) communication rounds per query. Each round incurs network latency. The servers must be online and communicating throughout every query.

In threshold CKKS, the data stays encrypted on a single server. All computation happens non-interactively using HE. Interaction only occurs once per query, at decryption time, with a single round-trip to the key committee.

The fundamental tradeoff: PRAG distributes trust over the data and computation (stronger but more expensive). Threshold CKKS distributes trust only over the decryption key (simpler, fewer rounds, but the DB server sees ciphertext access patterns).

Opaque's decoy mechanism mitigates the access pattern concern. The combination of HE computation (non-interactive, fast) with threshold decryption (one interactive round) and decoy-based pattern hiding provides a practical privacy profile with much lower latency than pure MPC.

For a research contribution, the interesting direction is formally comparing these approaches across security guarantees, latency, communication complexity, and scalability, and exploring whether HE search with MPC-based key management offers a better tradeoff than either pure approach.

## Implementation Roadmap

### Phase 1: Engine Refactor — DONE

Created `HEProvider` interface with `DirectHEProvider` and `ThresholdHEProvider` implementations. The client uses the interface, making the HE backend pluggable. Evaluator-only engines use committee eval keys without any secret key material.

### Phase 2: Threshold Decryption POC — DONE

Implemented full threshold CKKS using Lattigo's `mhe` package. Local POC where `Committee` simulates N participants in-process. Validated: collective key generation, Shamir threshold distribution, partial key-switch with noise flooding, correct results within expected tolerance, latency overhead benchmarked. ThresholdDecrypt is parallelized internally — Shamir-to-additive conversion and PCKS partial share generation run concurrently per participant. Each `thresholdEvalEngine` has its own decryptor/encoder for thread safety — Lattigo's `rlwe.Decryptor` is NOT thread-safe, and sharing a decryptor across goroutines caused underflow panics in concurrent decryption (data race). Noise flooding sigma=2^20 balances security vs precision.

### Phase 3: Committee gRPC Service

Implement a lightweight gRPC service for committee nodes. RPCs: PartialKeySwitch (per query), SetupRound (during collective key generation). The committee node binary should be minimal: load a key share, serve partial key-switch requests.

### Phase 4: Distributed Integration

Replace the local `Committee` with a network-distributed committee. The DB server coordinates between the client and remote committee nodes. Add configuration for committee endpoints, threshold t, and node discovery.

### Phase 5: Hardening

Implement redundancy-based bad node detection. Add metrics and logging for committee health. Benchmark with realistic network latencies. Stress test with node failures.

## Open Research Questions

1. Can the collective key generation be made non-interactive using a trusted setup ceremony (similar to zk-SNARK ceremonies)? This would simplify the setup phase.

2. What is the optimal (t, N) ratio for different deployment models? Higher t increases security but reduces availability and increases latency.

3. Can we combine threshold decryption with oblivious cluster selection (HE-native top-K) to eliminate the need for decoys entirely? This would require polynomial approximations of comparison functions within the CKKS noise budget.

4. How does the noise budget scale if we add threshold decryption to deeper circuits (e.g., HE-native centroid updates from the Tier 3 incremental indexing roadmap)?

5. Can verifiable partial decryption be made efficient enough for production use? Recent work on lattice-based commitments and short proofs may be applicable.

6. Is there a hybrid architecture where HE handles search and MPC handles only the key management and threshold decryption, combining the best of both worlds? This is the direction Guy Zyskind suggested and is potentially a novel research contribution.
